Потребители модулей! Разочарованы тем, что каждый модуль имеет свои собственные дико уникальный формат README? Раздражают модули, в которых отсутствуют важные разделы, такие как API или Примеры? Застрял, просматривая документацию по API еще до того, как узнал об этом что делает модуль?

Авторы модулей! Устали придумывать формат файла readme каждый раз, когда вы его пишете? Хотите, чтобы файлы readme были предварительно заполнены именем, описанием и лицензией вашего модуля, и вам не нужно каждый раз беспокоиться о структуре?

Что, если бы был общий формат для удобства производителей и потребителей?

Общий файл readme для модулей узлов.

Это может сэкономить время всем, если придерживаться 4 принципов:

  1. Никакой привязки. Никаких специальных форматов или инструментов; запустите common-readme один раз для чистой ванильного markdown.
  2. Никаких сюрпризов. Извлеките как можно больше деталей из package.json — например, имя, описание и лицензия. Не тратьте время на настройку.
  3. Когнитивная воронка. Начните с самой общей информации вверху (Имя, Описание, Примеры) и, если читатель сохраняет интерес, сузьте до конкретики (API, Установка). Это позволяет читателям "замкнуть" и продолжить поиск нужного модуля в другом месте, не тратя время на ненужные детали.
  4. Последовательность. Ваш мозг может сканировать документ гораздо быстрее, когда он может предвидеть его структуру.

Общий формат

common-readme работает по принципу когнитивной воронки.

В идеале тот, кто немного знаком с вашим модулем, должен иметь возможность освежить свою память, не нажимая "страница вниз". По мере того, как ваш читатель продолжает читать документ, он должен получать все больше и больше знаний. -- perlmodstyle

Использование

Установив npm, запустите

$ npm install -g common-readme

common-readme — это программа командной строки. Вы запускаете ее, когда запускаете новый модуль, в котором настроен package.json.

При запуске создается совершенно новый README, который записывается на ваш терминал. Вы можете перенаправить его в README.md и использовать его в качестве основы для вашего нового модуля.

$ common-readme > README.md

Этот совершенно новый файл readme будет автоматически заполнен значениями из package.json, такими как name, description и license. Для всего остального (использование, API и т. д.) будут созданы разделы-заглушки, готовые для заполнения.

Почему?

Это не безумная новая идея. Другие экосистемы, такие как Perl's CPAN, уже много лет пользуются общим форматом readme. Более того:

  1. Сообщество Node поддерживается нами, людьми, и модулями, которыми мы делимся. Это наша документация, которая связывает нас вместе. Наш README — это первое, что видят разработчики, и он должен быть максимально эффективным для передачи его цели и функции.

  2. Много мудрости можно почерпнуть у многих разработчиков, которые написали много-много модулей. Общий readme направлен на то, чтобы переработать этот опыт в общий формат, который принесет пользу всем нам; особенно новым разработчикам!

  3. Написание одного и того же шаблона — пустая трата времени каждого автора — мы могли бы также сгенерировать общие части и позволить автору сосредоточиться на содержании.

  4. Просмотр модулей на npm является частью обычного цикла разработки каждого разработчика узлов. Последовательный формат позволяет мозгу сосредоточиться на содержании, а не на структуре.
